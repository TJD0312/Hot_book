# AI 里的 Tools，到底是什么？

## 选题定位

本篇聚焦 `Tools`（工具）的入门理解，核心问题是：AI 里的 `Tools` 到底是什么？

面向第一次听说 `Tools`、`Tool Use` 或 `Function Calling` 的普通读者，不从 API 参数或平台实现讲起，而是解释一个更直观的问题：为什么一个只会生成文字的 `LLM`，需要通过工具去连接外部能力。

## 历史选题避让

- 已有 `MCP` 主题讲过“AI 怎么连接外部工具、数据和服务”，本篇不重复讲协议、`MCP Server`、`MCP Client` 或 USB-C 类比。
- 已有 `Agent 长任务失败` 主题提到工具调用可能出错，本篇不展开长任务误差累积。
- 已有 `Agent Memory` 主题讲记忆机制，本篇不讲长期记忆或状态保存。

## 核心观点

`Tools` 不是让模型本体突然拥有现实世界的能力，而是 AI 系统开放给模型的一组外部能力入口。

模型通常负责判断“现在是否需要调用某个工具、调用哪个工具、用什么参数”；真正执行工具的，是应用、后端服务或平台提供的运行环境。工具返回结果后，模型再把结果纳入上下文，继续生成回答或决定下一步。

## 推荐标题方向

1. AI 里的 Tools，到底是什么？
2. 别把 Tools 理解成“模型自己会做事”
3. 为什么 Agent 一定要有 Tools？
4. AI 调工具时，真正动手的是谁？

## 标题页建议

标题：AI 里的 Tools，到底是什么？

副标题：不是模型自己会做万事，而是它能调用外部能力

辅助文案：让 AI 调用外部能力

## 页面结构

1. 标题页：提出“Tools 到底是什么”的问题。
2. 第 1 页：只会生成文字，还不等于能做事。
3. 第 2 页：`Tools` 是外部能力入口。
4. 第 3 页：模型负责判断“要不要用工具”。
5. 第 4 页：真正执行工具的是系统，不是模型本体。
6. 第 5 页：工具结果会回到上下文里。
7. 第 6 页：工具能减少“靠猜”，但不能保证全对。
8. 第 7 页：`Tools` 和 `MCP` 不是一回事。
9. 第 8 页：越能做事，越要设计权限和确认。
10. 第 9 页：理解 `Tools` 的关键关系。

## 表达边界

- 不把 `Tools` 说成某一个平台独有能力。
- 不展开具体 API 参数，例如 `tools`、`tool_choice`、`tool_result` 的平台差异。
- 不承诺工具调用一定正确；工具本身、参数、权限、数据质量和结果校验都会影响可靠性。
- 不把 `Tools` 等同于 `MCP`。`Tools` 是可调用能力，`MCP` 是连接这些能力的一类标准方式。
