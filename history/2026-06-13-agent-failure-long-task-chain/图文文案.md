# 标题页

标题：
Agent 为什么会失败？

副标题：
不是模型笨，而是任务链太长

封面辅助文案：
每一步小错，都会在后面被放大

# 第 1 页

页面标题：
Agent 不只是回答一句话

页面正文：
`Agent` 通常不是只生成一段文字。

它会在任务循环里做事：

先理解目标，
再拆成步骤，
调用工具，
观察结果，
然后决定下一步。

任务越长，步骤越多。

步骤越多，出错点也越多。

# 第 2 页

页面标题：
真正危险的是“前错后继”

页面正文：
很多时候，Agent 不是突然在某一步崩掉。

它可能每一步看起来都还算合理：

读需求、拆任务、找资料、调工具、整理结果。

但如果第一步理解偏了一点，
后面的步骤就会沿着这个偏差继续往前走。

前一步的小错，
会变成后一步的输入。

# 第 3 页

页面标题：
小错是这样被放大的

页面正文：
比如你让 Agent 做一份资料整理。

第一步，它误解了你的目标。

第二步，它按错误目标去搜资料。

第三步，它用不完整资料做判断。

第四步，它把中间判断写成总结。

最后的结果可能很完整，
但方向已经偏了。

# 第 4 页

页面标题：
这就是多步任务误差累积

页面正文：
`LLM` 每一步都可能引入一点不确定性。

不确定性可能来自理解偏差、检索遗漏、工具参数写错，或对结果解释过度。

单看某一步，错误很小。

但在长任务里，
小错会被下一步当成事实继续使用。

# 第 5 页

页面标题：
所以 Agent 需要记住“状态”

页面正文：
这里说的 `Memory`（记忆机制），
不是让模型真的像人一样拥有记忆。

更重要的是记录关键状态：

目标是什么？
已经做过什么？
哪些假设还没确认？
哪里可能有风险？

否则任务做得越久，
Agent 越容易忘记一开始要解决什么。

# 第 6 页

页面标题：
校验，是给长任务加闸门

页面正文：
可靠的 Agent 不能只顾着往前走。

它需要 `Verification`（校验机制）：

当前步骤是否符合目标？
工具返回是否可信？
中间结论有没有证据？
输出是否满足约束？

最好不是等最后才检查，
而是在关键节点就拦住错误。

# 第 7 页

页面标题：
回滚，让 Agent 能退回重来

页面正文：
如果发现方向错了，
继续往前做只会增加成本。

`Rollback`（回滚机制）的作用，
就是让系统回到上一个可靠状态。

重新拆任务，
重新调用工具，
或者换一条策略。

它不等于让模型变聪明，
而是让系统不必一直错下去。

# 第 8 页

页面标题：
任务链越长，越需要控制偏差

页面正文：
Agent 做长任务会失败，
常常不是因为模型“太笨”。

而是因为任务链太长，
状态太多，
每一步的小错都可能传给下一步。

所以靠谱的 Agent，
不只是会执行。

还要会记住关键状态，
会及时校验，
也能在必要时回滚。

# 小红书正文文案

正文标题：
Agent 为什么会失败？不是模型笨，而是任务链太长

正文内容：
🧭 Agent 为什么会失败？

很多人会直接归因于“模型不够聪明”。

但在真实任务里，问题通常出在任务链上。

`Agent` 不是只回答一句话。

它要理解目标、拆步骤、调用工具、观察结果。

然后继续决定下一步。

🔍 多步任务最怕“误差累积”。

第一步目标偏了一点。

第二步就会按错误目标找资料。

资料不完整，后面的判断继续偏。

最后结果可能很完整。

但方向已经错了。

🛠️ 可靠的 Agent 系统需要三类机制：

`Memory` 记录目标、进度、假设和风险点。

`Verification` 在关键步骤检查证据和约束。

`Rollback` 发现方向错了，就退回可靠状态。

不是继续沿着错误路线往前做。

真正的难点不是“下一步答不答对”。

而是很多步之后还能不能不偏航。

靠谱的 Agent 不只是会执行。

还要会记住、会检查、会退回。
