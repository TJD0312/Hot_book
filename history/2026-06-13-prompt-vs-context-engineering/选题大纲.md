# Prompt Engineering 和 Context Engineering 到底差在哪？

## 选题定位

本篇面向普通 AI 使用者，聚焦一个入门问题：`Prompt Engineering`（提示词工程）和 `Context Engineering`（上下文工程）到底差在哪。

文章不做开发教程，不展开 API 参数、Agent 架构或具体平台能力。核心目标是帮助读者理解：AI 回答质量不只取决于“这句话怎么问”，还取决于模型回答前看到了哪些材料、规则、历史和状态。

## 核心观点

`Prompt Engineering` 更像把当前这次请求说清楚：目标是什么、输出成什么格式、语气如何、有哪些限制。

`Context Engineering` 更像整理模型回答前能看到的信息环境：相关资料、背景规则、历史偏好、任务状态、工具结果等。

二者不是对立关系。`Prompt` 是上下文的一部分，但上下文不只有 `Prompt`。当任务很简单时，写清楚 `Prompt` 往往就够；当任务需要长期稳定、结合资料、保持风格或延续偏好时，只改一句提示词通常不够。

## 表达边界

- 不把 `Context Engineering` 说成已经有完全统一定义的行业标准；它仍是快速发展的实践术语。
- 不说 `Prompt Engineering` 已经过时。更准确的说法是：它仍然重要，但不是全部。
- 不把上下文简化成“越多越好”。上下文需要相关、清楚、不过载、不互相矛盾。
- 不展开具体产品功能、价格、模型版本或 API 能力，避免时效性误导。
- 类比只用于辅助理解，不当作严格技术定义。
