# 资料核查

## 核查时间

2026-06-13

## 核心核查结论

- `Prompt Engineering` 可以概括为设计和优化给模型的指令，使模型更稳定地产生符合需求的输出。
- `Context Engineering` 是近年兴起的实践术语，常用于描述对模型上下文窗口中信息的选择、组织、压缩和管理。
- 二者不是对立关系。`Prompt` 是上下文的一部分，但上下文还可能包含系统规则、参考资料、对话历史、工具结果、任务状态和记忆信息。
- 本稿面向普通用户，不采用严格工程定义，而采用更易理解的表达：`Prompt Engineering` 偏“怎么问”，`Context Engineering` 偏“给模型看什么信息环境”。

## 主要资料来源

1. OpenAI API 文档：Prompt engineering  
   https://developers.openai.com/api/docs/guides/prompt-engineering  
   核查点：文档将 `Prompt engineering` 描述为为模型编写有效指令，使其生成符合要求的内容；也强调复杂应用中需要测试和评估 prompt 行为。

2. Anthropic 文档：Prompt engineering overview  
   https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/overview  
   核查点：文档强调在做提示词优化前，应先明确成功标准和测试方式；并指出不是所有失败指标都适合只靠 prompt engineering 解决。

3. LangChain Blog：Context Engineering  
   https://www.langchain.com/blog/context-engineering-for-agents  
   核查点：文章将 `Context Engineering` 概括为在 Agent 每一步轨迹中，为上下文窗口填入“刚好合适的信息”；并列出常见上下文类型，包括 instructions、knowledge 和 tools。

4. A Survey of Context Engineering for Large Language Models  
   https://arxiv.org/abs/2507.13334  
   核查点：论文将 `Context Engineering` 描述为超越简单 prompt design 的系统性优化，涵盖 context retrieval and generation、context processing 和 context management。

## 表达取舍

- 没有把 `Context Engineering` 定义为某个固定平台功能，因为该术语仍在快速发展，不同社区有不同侧重。
- 没有说 `Prompt Engineering` 已经过时。更稳妥的表达是：它仍然重要，但不足以覆盖所有 AI 使用和系统设计问题。
- 没有展开 `RAG`、`Agent Memory`、`Tools` 或 `MCP` 的细节，避免与历史选题重复。
- 使用“外卖下单”类比，只用于帮助非技术读者理解“需求表达”和“信息环境”的差别，不作为技术定义。
