# 资料核查

## 核查日期

2026-06-13

## 核查来源

1. MCP 官方介绍  
   https://modelcontextprotocol.io/docs/getting-started/intro

   核查要点：
   - MCP 是用于让应用向 `LLM` 提供上下文的一套开放协议。
   - 官方介绍中使用了类似 `USB-C` 的类比，用来说明 MCP 为 AI 应用连接数据源和工具提供标准方式。
   - MCP 面向的不只是单一工具，而是外部数据源、工具和工作流等能力。

2. MCP 官方架构说明  
   https://modelcontextprotocol.io/docs/learn/architecture

   核查要点：
   - MCP 架构中有 `MCP Host`、`MCP Client` 和 `MCP Server`。
   - `MCP Host` 是用户直接使用的 AI 应用，例如聊天工具、IDE 或其他 AI 程序。
   - `MCP Client` 通常由 host 创建，用来和某个 server 保持连接。
   - `MCP Server` 暴露外部能力，供 client 发现和调用。

3. MCP 官方规范页面  
   https://modelcontextprotocol.io/specification/2025-06-18

   核查要点：
   - MCP 中常见能力包括 `Tools`（工具）、`Resources`（资源）和 `Prompts`（提示模板）。
   - 实际能力发现、调用、权限和安全边界需要由具体应用和 server 实现共同保证。

## 本篇采用的科普表达

- 把 MCP 解释为“让 AI 应用连接外部工具、数据和服务的一套标准规则”。
- 使用“AI 的 USB-C”作为类比，强调“连接方式更统一”，不把它当作严格技术定义。
- 用 `MCP Server` 和 `MCP Client` 解释基本连接关系，但不展开协议报文、传输层或 JSON-RPC 细节。

## 需要避免的说法

- “MCP 是一个新模型”：不准确。
- “MCP 等于 RAG”：不准确。`RAG` 更偏检索增强生成，MCP 是连接协议。
- “有了 MCP，AI 就能自动使用所有工具”：过度简化。工具需要 server 暴露能力，host/client 需要支持协议，权限和安全也需要配置。
- “接入 MCP 后结果一定可靠”：不准确。工具数据质量、权限设计、调用结果校验仍然重要。
