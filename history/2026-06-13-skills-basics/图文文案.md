# 标题页

标题：
`Skills` 到底是什么？

副标题：
不是连接工具，而是教 Agent 怎么做事

辅助文案：
操作手册、示例和约束，帮 Agent 少靠临场猜

# 第 1 页

页面标题：
先别把 `Skills` 理解成 `MCP`

页面正文：
很多人第一次听到 `Skills`，会自然联想到 `MCP`。

但它们解决的不是同一个问题。

`MCP` 更像一套连接规则，帮助 AI 应用连接外部工具、数据和服务。

`Skills` 更像一套做事方法，告诉 Agent 遇到某类任务时应该怎么处理。

# 第 2 页

页面标题：
`MCP` 解决的是“怎么接上”

页面正文：
如果 AI 应用想读取文件、查询数据库、调用业务系统，就需要一种方式和外部能力连接。

`MCP` 的重点，是让这些连接更统一、更可复用。

可以简单理解为：

它关心的是“外部能力怎样被发现、读取和调用”。

# 第 3 页

页面标题：
`Skills` 解决的是“怎么做好”

页面正文：
接上工具之后，Agent 仍然要知道：

这类任务先做哪一步？

中间要检查什么？

结果应该整理成什么格式？

哪些行为不能做？

这些“做事方法”，就是 `Skills` 最重要的价值。

# 第 4 页

页面标题：
`Skill` 像一份操作手册

页面正文：
一个 `Skill` 通常会告诉 Agent：

什么场景下该使用它。

应该按什么步骤完成任务。

可以参考哪些示例。

遇到风险时要遵守哪些约束。

它不是让模型突然更聪明，而是把做事流程提前整理好。

# 第 5 页

页面标题：
为什么 Agent 需要它？

页面正文：
没有 `Skills` 时，Agent 很多时候要临场判断流程。

同样是“做一份 PPT”，它可能每次都用不同方式理解任务。

有了 `Skills`，系统可以提前告诉它：

先确认主题，再拆结构，再写页面，再检查版式。

这样任务路径会更稳定。

# 第 6 页

页面标题：
它更像“经验复用”

页面正文：
`Skills` 的价值，不只是写几条提示词。

它可以把一类任务中反复出现的经验沉淀下来：

读论文时先看摘要和图表。

处理表格时先检查表头和缺失值。

做图文时先确认选题，再写文案，最后才生成图片。

# 第 7 页

页面标题：
`Tools` 和 `Skills` 差在哪？

页面正文：
`Tools` 更像“能调用什么”。

比如搜索、读取文件、生成图片、操作表格。

`Skills` 更像“怎么把这些能力用好”。

比如什么时候该搜索，搜索后怎么核查，什么时候不能直接生成最终结果。

一个偏能力入口，一个偏任务方法。

# 第 8 页

页面标题：
`MCP` 和 `Skills` 差在哪？

页面正文：
可以用一个简单关系理解：

`MCP`：让 Agent 更容易接上外部能力。

`Tools`：外部能力本身可以被调用。

`Skills`：告诉 Agent 面对某类任务时，怎么组织步骤和约束。

所以，`MCP` 管连接，`Skills` 管方法。

# 第 9 页

页面标题：
好 `Skill` 不是万能答案

页面正文：
`Skills` 能让 Agent 更稳定，但不能保证永远正确。

如果操作手册写得含糊，示例过时，约束不清楚，Agent 仍然可能做错。

所以理解 `Skills` 的关键是：

它不是替代判断，而是把可靠做法提前写进任务流程里。

# 小红书正文文案

正文标题：
`Skills` 到底是什么？别再把它和 `MCP` 混在一起

正文内容：
很多人听到 AI Agent 里的 `Skills`，会以为它和 `MCP` 差不多，都是“让 AI 多一些能力”。

但更准确地说，它们解决的是两个不同问题。

`MCP` 更像连接规则，重点是让 AI 应用更统一地连接外部工具、数据和服务。

`Skills` 更像操作手册，重点是告诉 Agent 遇到某类任务时，应该按什么步骤做、参考哪些示例、遵守哪些约束。

比如做 PPT、读论文、处理表格、生成图文，这些任务不只是“能不能调用工具”的问题，还涉及流程、检查点、输出格式和边界。

所以可以这样理解：

`Tools` 关心“能调用什么”。

`MCP` 关心“怎么连接这些能力”。

`Skills` 关心“怎么把这类任务做稳”。

好的 `Skill` 不是让模型本体突然更聪明，而是把一套可复用的做事方法提前整理好，让 Agent 少靠临场猜。
